### 2. JSON-LD pretty-print removal (`null, 2` dropped) — XSS protection intact
**Severity: No Finding**

**What changed:**
```ts
// src/lib/seo/structured-data.ts  (stringifyJsonLd)
// Before:
JSON.stringify({ '@context': '...', ...data }, null, 2).replace(/</g, '\\u003c')

// After:
JSON.stringify({ '@context': '...', ...data }).replace(/</g, '\\u003c')
```

**Analysis:**
The removal of `null, 2` (indentation arguments) changes only whitespace formatting in the output. The security-critical `.replace(/</g, '\\u003c')` chain is preserved in the same position and still executes after `JSON.stringify`. The XSS protection mechanism is unchanged.

The escape of `<` to `\u003c` is the standard defence against the primary HTML injection vector in JSON-LD `<script>` blocks. The classic attack requires `</script>` to close the script tag — once `<` is replaced, that sequence cannot form and no script tag can be closed or opened.

Verified by runtime test:
```
Input:  </script><script>alert(1)</script>
Output: \u003c/script>\u003cscript>alert(1)\u003c/script>
```
The attack string is fully neutralised.

**Secondary consideration — unescaped `>` characters:**
A lone `>` character cannot close or open an HTML tag without a preceding `<`. Inside a `<script type="application/ld+json">` block, `>` in a JSON string value is inert to the HTML parser. This is safe.

**Secondary consideration — U+2028 / U+2029 (Line Separator / Paragraph Separator):**
`JSON.stringify` in Node.js 20 does NOT escape these characters (they are emitted as raw UTF-8 bytes). In a `<script type="text/javascript">` block they would be dangerous because they are valid JS line terminators. However, this output is placed in `<script type="application/ld+json">` blocks only — those blocks are parsed as JSON data by the browser's JSON-LD processor, not executed as JavaScript. U+2028/2029 are valid within JSON string values per RFC 8259, so there is no injection risk in this specific context.

**Conclusion:** Removing pretty-printing does not weaken XSS protection. The change is safe.

## Observations and Recommendations (Non-blocking)

### Rec 1: Add `>` escaping to `stringifyJsonLd` for belt-and-suspenders hardening
The current escape of `<` is sufficient for preventing `</script>` injection. Adding `>` escaping is not required but is a common belt-and-suspenders practice used by Google's own structured data helpers:

```ts
.replace(/</g, '\\u003c')
.replace(/>/g, '\\u003e')
.replace(/&/g, '\\u0026')
```

This would also cover edge cases where a future schema type introduces content from a CMS or database.

### Rec 2: Consider escaping U+2028 / U+2029 if schema data ever becomes dynamic
If `createFAQPageSchema` or `createBlogPostingSchema` are ever extended to accept CMS-sourced content (not hardcoded strings), add:

```ts
.replace(/\u2028/g, '\\u2028')
.replace(/\u2029/g, '\\u2029')
```

This is a precautionary measure for future-proofing; it is not required for the current static data sources.

### Rec 3: `SECURITY_EMAIL` not linked in the rendered CTA box
The bottom CTA on `src/app/security/page.tsx` (lines 71-77) uses `CONTACT_EMAIL` for both the email href and display text, even though the page now advertises `SECURITY_EMAIL` in the "Responsible disclosure" card. Consider whether the CTA box should also offer a direct `mailto:${SECURITY_EMAIL}` link for researchers.
